# Vicode 0.2.1 Reviewer Guide

This guide is for someone reviewing Vicode for the first time on Windows.

It explains:

- what Vicode is
- how beta testers should install it
- how to run it locally from source when needed
- how to build the Windows installer and app
- what the current npm story is
- what the current beta limits are

## What Vicode Is

Vicode is a Windows-first Electron desktop app for AI-assisted coding work.

It is intentionally smaller than Codex or full IDE shells. The current release
line focuses on:

- local project and thread workflows
- trustworthy provider execution
- a calm transcript-based desktop UI
- packaged Windows beta delivery

## Current Release Posture

For `0.2.1`, the current external beta path is the packaged Windows app:

- `release/Vicode-Setup-0.2.1.exe`
- `release/win-unpacked/Vicode.exe`

Installed Windows builds now check GitHub Releases for desktop updates on
launch, show update state in `Settings > General`, and surface a visible
titlebar update action when a new build is ready.

If a run is active when the user clicks update, Vicode queues the install and
waits for that run to finish before restarting.

Those desktop update artifacts publish to the public release-only repo
`lowluds/vicode-windows-releases`.

Public beta source snapshots should be published separately through the curated
mirror repo `lowluds/vicode-windows-public` so the internal working repo and
its history do not need to be exposed directly.

`npm pack` is available for audit and internal inspection, but `npm publish` is
still not the supported end-user install route for this release line.

## Beta Tester Quick Start

For a first-time beta tester, use this path:

1. Download the installer from `lowluds/vicode-windows-releases`.
2. Install and launch Vicode on Windows.
3. Trust a project folder.
4. Connect `OpenAI / Codex CLI`, `Gemini CLI`, or `Ollama`.
5. Use the titlebar update action or `Settings > General` when an update is ready.

## Requirements

Recommended local baseline:

- Windows 11
- Node.js `24.x`
- npm `10+`

Install dependencies:

```bash
npm install
```

If Electron native dependencies drift after install or after an Electron
upgrade, repair them with:

```bash
npm run native:prepare:electron
```

## Run The App Locally

Start the desktop app in development mode:

```bash
npm run dev
```

Use this when you want to inspect the live UI directly while developing.

## Test The Current Desktop Build

The canonical local desktop sweep is:

```bash
npm run verify:desktop
```

That runs:

```bash
npm run build
npm test
npm run native:prepare:electron
npm run smoke
npm run e2e
```

If you want the smaller baseline first, run:

```bash
npm test
npm run build
npm run smoke
```

## Build The Windows Installer And Desktop App

Create the packaged Windows artifacts:

```bash
npm run dist:win
```

Then validate the packaged output:

```bash
npm run smoke:packaged
npm run audit:release
npm run audit:repo-public
```

Publish the Windows update channel only when the candidate is backed by a tag
or draft release in `lowluds/vicode-windows-releases`:

```bash
npm run release:win
```

Manual packaged beta certification is still required before external handoff:

- install the NSIS installer
- confirm first launch and relaunch
- confirm uninstall
- confirm hashes and signer status for the candidate

## Current Beta Reality

Current repo-local beta truth:

- core build, unit-test, smoke, packaged-smoke, release-audit, and public-doc
  audit commands are part of the beta readiness path
- the packaged Windows app is the supported install and update surface
- npm packaging remains an internal audit lane, not the supported public install
  route
- desktop auto-update is wired, but it still needs a real published release
  cycle check against a newer GitHub release before it should be called fully
  update-certified
- broad public launch claims still require more hardening than the current
  invite-style beta

## Current NPM Story

`npm pack` currently produces a package tarball for inspection.

That is not the same as saying the npm package is ready for public end users.

The current tarball still fails a clean `npm install` because the package
postinstall path expects `electron`, while the repo's Electron Builder setup
requires `electron` to remain in `devDependencies`.

For `0.2.1`, use this rule:

- packaged Windows app is the supported reviewer, beta, and installed-app
  update path
- npm packaging remains an audit lane, not the primary installation path
